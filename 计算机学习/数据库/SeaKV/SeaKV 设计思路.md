目前想做一个键值对存储系统，从这里开始写一个思路，需求如下：
- 读写性能平衡，因此选择 LSM-Tree 作为存储引擎。
- 需要能够应对多点故障的高可靠性和强一致性，选择 Raft 共识算法作为数据同步的状态机。
- 为方便调试和兼容主流的 KV 客户端生态，选择支持 redis 的协议。
- 网络 I/O 可以优化的空间有限，可以选择 go net，也可以选择 gnet，两者在编程过程中区别不大，但是底层 gnet 的 epoll 使用了 ET 模型，并且还做了一些其他方面的优化，所以在某些方面表现可能更好。
- 磁盘 I/O 可以选择 io_uring 进行优化，但是对于非 linux 平台我不太了解其解决方案，可以选择直接维护一个专门用于 I/O 操作的协程工作池。
- 减少锁的使用，可以选择实现 MVCC 来实现高效的读写。

## 总体架构

外部通过 redis 协议进行访问，gnet 负责处理网络 I/O，这里肯定是多线程，分发命令交给 raft 状态机，然后 raft 状态机交给底层的存储引擎进行处理。

需要抽象的部分有：
1. 网络 I/O 事件分发访问存储引擎通过接口访问。
2. 存储引擎通过接口来访问接口来访问文件系统。

## 开发

### MVP 阶段

从上至下慢慢开发，存储引擎先用 go 的 map 来进行替代，最开始先将 redis 协议和网络 I/O 这块写好，就算是 map，也要先定义好其接口实现，方便后续针对于接口进行存储引擎的开发。（此时还是单机数据库，可以选择同时让 map 存储引擎也支持 raft 同步）

### 存储引擎

开发 lsm-tree 存储引擎，根据 mvp 阶段的存储引擎接口定义进行开发，此时可以直接将 raft 状态机依赖的存储引擎替换，数据存储可以先用内存存储，并且定义一个文件存储的接口，方便后续开发基于文件或者云存储的实现，也方便单测。

### 优化阶段

考虑使用协程工作池或者 io_uring 来优化 I/O，参考 rocksdb、NoKV、tiny-lsm-go 可以进行 prefetch、布隆过滤器等等的优化。
